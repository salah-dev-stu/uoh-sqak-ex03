\chapter{Production Patterns and Challenges}\label{ch:production}

Production multi-agent systems operate at scales that demand explicit attention to cost, reliability, and observability. This chapter explores the operational patterns that distinguish experimental prototypes from production-grade orchestrated systems.

\section{Rate Limiting and Token Budget Management}

Token consumption in multi-agent pipelines grows nonlinearly with agent count and task complexity. An uncontrolled system can exhaust its budget in a single failed orchestration run, leaving no tokens for recovery or exploration. Budget management requires two mechanisms: allocation and enforcement.

\subsection{Token Budget Allocation Formula}

The total token budget must be distributed across agents with reserves for retries and unexpected expansion:

\begin{equation}
B_{\text{total}} = \sum_{i=1}^{N} B_i + B_{\text{overhead}}
\label{eq:token_budget}
\end{equation}

where $B_i$ is the quota for agent $i$, $N$ is the agent count, and $B_{\text{overhead}}$ is a buffer (typically 10\% of the main budget) reserved for failed retries. A typical distribution for a four-agent system (Researcher, Writer, Editor, LaTeX Producer) allocates:
\begin{itemize}
  \item Researcher: 40\% (high due to web search and source synthesis)
  \item Writer: 25\% (narrative generation)
  \item Editor: 20\% (quality control loop)
  \item LaTeX Producer: 10\% (final compilation validation)
  \item Overhead reserve: 5\%
\end{itemize}

These quotas must live in configuration files (JSON or YAML), never hardcoded in source code. Configuration must specify:
\begin{itemize}
  \item \texttt{max\_tokens\_per\_agent}: hard ceiling for each agent invocation
  \item \texttt{global\_rate\_limit}: tokens per minute across all agents
  \item \texttt{circuit\_breaker\_threshold}: failure count before pausing
  \item \texttt{token\_warning\_threshold}: percentage (e.g., 80\%) at which alerts fire
\end{itemize}

\cite{Anthropic2024b} emphasizes that configuration portability is a prerequisite for production systems: the grading infrastructure must be able to clone a repository and run the system on a fresh machine with identical behavior. Hardcoded token limits, API keys, or rate parameters prevent this.

\section{Cost Optimization and Tracking}

Beyond tokens, financial cost must be explicitly tracked and bounded. Cost depends on both tokens and model selection. The cost formula is:

\begin{equation}
C = \sum_{k=1}^{K} \bigl( c_{\text{in}} \cdot t_{k}^{\text{in}} + c_{\text{out}} \cdot t_{k}^{\text{out}} \bigr)
\label{eq:cost}
\end{equation}

where $K$ is the total number of API calls, $t_k^{\text{in}}$ is input tokens for call $k$, $t_k^{\text{out}}$ is output tokens, $c_{\text{in}}$ is the per-1M-token input price, and $c_{\text{out}}$ is the per-1M-token output price. This cost function must be evaluated after \emph{each} agent invocation, with running totals logged. If cost exceeds a budget threshold, the system must gracefully degrade: return the best-effort result (e.g., incomplete research or rough draft) rather than failing silently \cite{CrewAI2024}.

\section{The Gatekeeper Pattern}

All external API calls must flow through a single Gatekeeper class. The Gatekeeper enforces:
\begin{itemize}
  \item Rate limiting: request queuing if rate limit would be exceeded
  \item Token accounting: every API call updates the running token count
  \item Retry logic: exponential backoff on transient failures
  \item Circuit breaking: pause an API endpoint if failure rate exceeds threshold
  \item Timeout enforcement: kill requests that exceed maximum duration
\end{itemize}

The Gatekeeper is the \textbf{only} place in the codebase where an external API is invoked. No agent directly calls an LLM; all calls are mediated. This centralization enables global observability: log lines from the Gatekeeper form a complete audit trail of system behavior.

\section{Retry and Circuit Breaker Patterns}

Transient failures (network timeouts, temporary API unavailability) are common in multi-agent systems. Blind retries amplify cost; strategic retries with backoff and circuit breaking provide resilience.

\subsection{Exponential Backoff}

When an API call fails, retry with exponential delays:

\begin{equation}
\text{delay}_n = \text{base\_delay} \times 2^{n} + \text{jitter}
\label{eq:backoff}
\end{equation}

where $n$ is the retry attempt number (starting at 0), and jitter is a random offset to prevent thundering herd. Typical parameters: base\_delay = 1 second, max retries = 4, jitter = 0--1 second. After the maximum retry count is exhausted, the system escalates: either fail the task (if critical) or return a fallback result (if non-critical).

\subsection{Circuit Breaker}

If an agent fails repeatedly within a time window (e.g., 5 failures in 5 minutes), the circuit breaker \textbf{pauses} further invocations of that agent for a cooldown period (e.g., 5 minutes). This prevents cascading failures: a broken downstream API (e.g., a web search service) no longer drowns the system in repeated failed requests. After cooldown expires, the circuit breaker enters a \textit{half-open} state, allowing one trial request. If the trial succeeds, the circuit resets; if it fails, the cooldown restarts.

\cite{LangGraph2024} demonstrates circuit breaker patterns in multi-agent graphs: failures at one node should not propagate to downstream nodes if the failure is transient.

\section{Observability and Structured Logging}

Production systems are opaque without detailed logging. Structured logging (JSON, not free-form text) enables automated analysis, alerting, and debugging.

Every significant event must be logged with a consistent schema:
\begin{itemize}
  \item \textbf{timestamp}: ISO 8601 format (e.g., \texttt{2026-06-09T14:32:51.123Z})
  \item \textbf{level}: DEBUG, INFO, WARNING, ERROR
  \item \textbf{component}: which module or agent generated the event
  \item \textbf{event\_type}: standardized event class (e.g., \texttt{task\_start}, \texttt{task\_complete}, \texttt{api\_call}, \texttt{error})
  \item \textbf{duration\_ms}: elapsed time for the operation
  \item \textbf{tokens\_consumed}: input and output token counts
  \item \textbf{cost\_usd}: financial cost of the API call
  \item \textbf{message}: human-readable description
\end{itemize}

Alert thresholds should be configured:
\begin{itemize}
  \item Agent response time \(> 30\) seconds: warning
  \item Task failure rate \(> 5\%\) in a 5-minute window: alert
  \item Token consumption \(> 90\%\) of allocated budget: escalate to high priority
  \item API rate limit hit (HTTP 429): immediate backoff
  \item Unhandled exception in agent loop: critical alert
\end{itemize}

Logs are the primary debugging artifact. When a production run fails, engineers analyze logs to reconstruct the exact sequence of events, API responses, and token consumption. Logs must be immutable, timestamped, and retained for at least 30 days \cite{Anthropic2024b}.

\section{Configuration Management and Versioning}

All mutable operational parameters must reside in configuration files, never embedded in source code. A typical configuration structure:

\begin{verbatim}
{
  "version": "1.00",
  "agents": {
    "researcher": {
      "rate_limit_rpm": 20,
      "token_quota": 5000,
      "timeout_seconds": 300,
      "model": "claude-opus-4-8"
    },
    "writer": {
      "rate_limit_rpm": 15,
      "token_quota": 3000,
      "timeout_seconds": 180
    }
  },
  "logging": {
    "level": "INFO",
    "output_path": "logs/crew.jsonl"
  },
  "rate_limits": {
    "global_rpm": 60,
    "circuit_breaker_threshold": 5,
    "circuit_breaker_window_seconds": 300,
    "circuit_breaker_cooldown_seconds": 300
  }
}
\end{verbatim}

Version numbers must increment with every code change. Semantic versioning is followed: \texttt{1.00} (initial), \texttt{1.01} (patch), \texttt{2.00} (breaking change). The version is stored in both \texttt{pyproject.toml} and the configuration JSON. This enables precise reproduction: an engineer can checkout a specific git commit and know that the corresponding configuration file has the same version number.

\section{Scaling Considerations}

As orchestrated systems grow from experimental to production scale, bottlenecks emerge:

\begin{itemize}
  \item \textbf{Orchestrator becomes a bottleneck}: if a single process sequences all agent invocations, latency grows linearly with agent count. Solution: deploy multiple Orchestrator replicas with a message queue (e.g., Kafka, RabbitMQ) to fan out tasks.
  
  \item \textbf{Token accounting overhead}: tracking tokens in-memory is fast; persisting to a database on every call is slow. Solution: batch token updates, flushing every 60 seconds or when a threshold is reached.
  
  \item \textbf{Configuration hot-reload}: redeploying the entire system to change a rate limit is operationally expensive. Solution: push configuration to a centralized service (e.g., etcd, Consul) and reload on-the-fly without restarting agents.
\end{itemize}

Production multi-agent systems demand engineering rigor: explicit budgeting, centralized gating, detailed observability, and configuration management. These patterns separate toy prototypes from systems trusted in critical workflows \cite{CrewAI2024}.
