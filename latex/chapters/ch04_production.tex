\chapter{Production Patterns and Challenges}\label{ch:production}

Transitioning from proof-of-concept to production requires three observability pillars without which systems cannot be safely deployed \cite{Segal2026}. First, \textbf{Full Trace} logs input, decision, tool invocation, output, cost, execution time, and action result for every agent step. Without trace, no accountability exists when failures occur. Second, \textbf{Continuous Evaluation} measures quality through automated tests—hallucination detection, policy adherence checks, reliability metrics across many runs. Third, \textbf{Human-in-the-Loop} enforcement ensures sensitive actions (send message, delete resource, modify permissions) require human approval before execution. Without Trace there is no accountability; without Evaluation there is no improvement; without Policy there is no safe production \cite{Segal2026}.

\section{Rate Limiting and Token Budgets}

Token economy fundamentally constrains multi-agent systems. Scaling a crew of $N$ agents processing $M$ articles through $K$ review cycles yields exponential resource consumption. Consider a five-agent crew (Researcher, Writer, Editor, Checker, Formatter) processing ten articles through three quality-review cycles: total execution count equals $5 \times 10 \times 3 = 150$ LLM calls. If each call averages 4{,}000 tokens combined (input plus output), total consumption reaches 600{,}000 tokens—substantial cost even at single-digit cents per million tokens \cite{Segal2026}.

Rate limiting prevents runaway consumption when agents loop or when topic complexity triggers excessive retrieval steps. A production system must define a \textbf{token budget} as a hard ceiling per agent, per day, or per month. The budget enforces discipline: when an agent approaches its limit, it must either summarize existing context or escalate to human review.

\begin{equation}
B_{\text{total}} = \sum_{i=1}^{N} B_{i} + R
\label{eq:token_budget}
\end{equation}

where $B_{\text{total}}$ is the monthly token allocation across $N$ agents, $B_{i}$ is the per-agent monthly budget, and $R$ is a reserve pool for unexpected high-complexity tasks. For a 10{,}000{,}000-token monthly subscription, distributing 1{,}000{,}000 tokens per agent (ten agents) plus 9{,}000{,}000 reserved for batch operations ensures predictable cost while maintaining flexibility \cite{CrewAI2024}.

\section{Gatekeeper Pattern}

All external API calls—to language models, vector databases, web search services, and LaTeX compilers—must flow through a single \textbf{Gatekeeper} class. This pattern centralizes three concerns: (1) rate limit enforcement, (2) credential management, (3) cost accounting.

The Gatekeeper intercepts every call. Before dispatch, it checks whether the requesting agent has remaining budget. If budget is exhausted, it raises an exception rather than silently failing. This explicit failure mode allows the orchestrator to escalate to human review or queue the task for later processing when budget replenishes. After dispatch, the Gatekeeper logs the actual token consumption and deducts it from the agent's available budget.

A typical Gatekeeper implementation exposes three methods: \texttt{check\_budget(agent\_id, estimated\_tokens)}, \texttt{record\_usage(agent\_id, actual\_tokens)}, and \texttt{get\_remaining(agent\_id)}. No agent code calls external services directly; all calls route through these Gatekeeper methods. This design prevents credential leaks (all keys stored in the Gatekeeper's secure configuration), enables fine-grained access control (disable a specific agent without code changes), and provides the audit trail required for compliance \cite{Segal2026}.

\section{Retry and Circuit Breaker}

Transient failures—network timeouts, rate limits from upstream services, temporary LLM unavailability—are inevitable in production. Two patterns mitigate them without infinite retry loops.

A \textbf{Retry} policy with exponential backoff re-attempts a failed call with increasing delays. A first failure triggers an immediate retry; a second triggers a 2-second delay before the next attempt; a third triggers an 8-second delay. After a configurable maximum (typically 3–5 retries), the system stops and escalates to a human or logs an alert.

A \textbf{Circuit Breaker} monitors downstream service health. When a service begins returning errors (failure rate exceeds a threshold), the circuit breaker opens: subsequent calls fail immediately without attempting to reach the service. This prevents cascading failures—the client stops hammering a failing service, allowing it time to recover. Once the service stabilizes, the circuit breaker transitions to half-open state, allowing a single probe request. If the probe succeeds, the circuit closes and normal operation resumes \cite{Segal2026}.

The retry probability for a task with $P_f$ failure rate per attempt and maximum $R$ retries is bounded:

\begin{equation}
P_{\text{success}} = 1 - P_f^{R+1}
\label{eq:retry_probability}
\end{equation}

For example, if a service has a 5\% failure rate ($P_f = 0.05$) and we allow 3 retries ($R = 3$), the probability of at least one successful attempt is $1 - 0.05^4 \approx 0.9999$—nearly certain success.

\section{Observability and Logging}

Production systems must emit comprehensive traces at every decision point. The trace includes:

\begin{itemize}
\item Agent name, task ID, start timestamp
\item Input context (truncated to first 500 chars for privacy)
\item Tool name and arguments
\item Tool response and response time
\item Agent reasoning and output
\item Token counts: input tokens, output tokens, cumulative
\item Cost in USD (model-dependent)
\item Human annotations (approval status, feedback tags)
\end{itemize}

CrewAI's callback system enables this through hook methods: \texttt{on\_task\_start}, \texttt{on\_task\_end}, \texttt{on\_tool\_use}, \texttt{on\_tool\_error}. An observer class subscribes to each hook, logging events to a FIFO buffer that is periodically written to disk. This buffer-and-flush pattern prevents logging overhead from dominating execution time.

Cost auditing allows accounting teams to allocate spend to projects and clients accurately, preventing budget surprises.

\section{Cost Optimization}

Three mitigation strategies reduce token consumption 40–60\%. \textbf{Context Reuse} passes agent output directly as context for subsequent agents, avoiding redundant re-encoding of intermediate results. CrewAI's context parameter automates this pattern. Instead of the Writer Agent re-reading the raw research topic and inferring background, it receives the Researcher Agent's synthesized notes as context—fewer tokens needed to reach the same quality.

\textbf{Rate Limiting} through a Gatekeeper enforces daily or monthly token budgets, preventing runaway consumption. When an agent approaches its limit, it must summarize or defer—no silent failures allowed.

\textbf{Model Selection} trades latency for cost: Claude Haiku executes substantially faster than Opus while consuming fewer tokens per call, suitable for high-volume, high-complexity tasks. For an article generation pipeline, the Researcher agent might use Opus (superior reasoning for complex topics), while the Editor agent uses Haiku (pattern-matching for grammar, lower cost).

The total production cost per article is modeled:

\begin{equation}
C = \sum_{k=1}^{K}\bigl(c_{\mathrm{in}}\,t_{k}^{\mathrm{in}} + c_{\mathrm{out}}\,t_{k}^{\mathrm{out}}\bigr)
\label{eq:cost}
\end{equation}

where $K$ is the number of LLM calls, $c_{\mathrm{in}}$ and $c_{\mathrm{out}}$ are per-token costs for input and output (model and provider dependent), and $t_k^{\mathrm{in}}$ and $t_k^{\mathrm{out}}$ are token counts for call $k$. For ten articles using Opus at a 2x/1x input/output ratio and a conservative token budget of 3{,}000 input and 2{,}000 output per call, total cost per article is approximately \$0.15–\$0.30. Scaling to 1{,}000 articles per month requires careful budget allocation and model selection discipline.

Architectural investment in rate limits, circuit breakers, and retry logic reduces operational friction during scaling. A system designed for 10 articles with 10{,}000{,}000 tokens will falter at 1{,}000 articles without guardrails; proper instrumentation enables 10x or 100x growth with confidence \cite{CrewAI2024}.
