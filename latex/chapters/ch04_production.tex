\chapter{Production Patterns and Challenges}\label{ch:production}

Multi-agent systems deployed to production face constraints absent in research prototypes: token budgets, rate limits, fault cascades, and cost overruns. This chapter examines five critical patterns for building reliable, cost-efficient orchestration systems \cite{Segal2026}.

\section{Rate Limiting and Token Budgets}\label{sec:ratelimit}

Every API call to an LLM incurs token costs and rate-limit exposure. A naive multi-agent system invoking agents in parallel without budget awareness can rapidly exhaust both quota and budget. Production systems must track token consumption per agent, enforce budget ceilings, and degrade gracefully when limits approach.

Define the token budget formula as follows. For a system with $K$ agents, each agent $k$ receives a fixed allocation. The total token spend across all agents is:

\begin{equation}
T_{\mathrm{total}} = \sum_{k=1}^{K}\bigl(t_{k}^{\mathrm{in}} + t_{k}^{\mathrm{out}}\bigr)
\label{eq:tokenbudget}
\end{equation}

where $t_{k}^{\mathrm{in}}$ is input tokens for agent $k$ and $t_{k}^{\mathrm{out}}$ is output tokens. Each agent must declare its maximum allocation upfront. If actual consumption $T_{\mathrm{total}}^{\mathrm{actual}} > B_{\max}$, where $B_{\max}$ is the global budget, the orchestrator halts new agent invocations and escalates to fallback strategies: switch to a cheaper model, simplify task requirements, or escalate to human review \cite{LangChain2024}.

Rate limits impose a second constraint. API providers enforce limits per minute or per day. The orchestrator must:

\begin{enumerate}
\item Track request volume per time window per API key
\item Queue excess requests instead of rejecting them
\item Implement backpressure: pause new work submission if queue depth exceeds threshold
\item Monitor quota depletion and alert at 75\% utilization
\end{enumerate}

\section{Gatekeeper Pattern}\label{sec:gatekeeper}

A \textbf{gatekeeper} is a singleton component that mediates all external API calls. Rather than agents calling LLM APIs directly, agents invoke the gatekeeper, which enforces policy: token budgets, rate limits, authentication, cost tracking, and audit logging.

The gatekeeper maintains two critical state objects:

\begin{itemize}
\item \textbf{Token ledger}: a per-agent counter incremented on every API call
\item \textbf{Rate-limit state}: per-API-key request timestamps for the current time window
\end{itemize}

When an agent requests an API call, the gatekeeper checks:

\begin{enumerate}
\item Is agent $k$'s token budget exhausted?
\item Has the API key hit its rate limit?
\item Is the request well-formed (non-null inputs, valid parameters)?
\end{enumerate}

If any check fails, the gatekeeper returns an error response with a suggested retry delay. If all checks pass, the gatekeeper calls the API, logs the result (tokens used, latency, status), and returns the response to the agent. This centralized enforcement prevents agents from circumventing policy \cite{CrewAI2024}.

\section{Retry and Circuit Breaker}\label{sec:retry}

Transient failures are inevitable in production: network timeouts, temporary rate-limit hits, brief service outages. A naive retry strategy risks amplifying failures. A robust strategy uses \textbf{exponential backoff} and \textbf{circuit breakers}.

\subsection{Exponential Backoff}

When an API call fails with a retryable error (HTTP 429 rate-limit, HTTP 503 service unavailable), the orchestrator retries after a delay that grows exponentially. The delay after attempt $i$ is:

\begin{equation}
\text{delay}_i = \min\bigl(b \cdot 2^i, d_{\max}\bigr) + \text{jitter}
\label{eq:backoff}
\end{equation}

where $b$ is the base delay (e.g., 1 second), $2^i$ is the exponential factor, $d_{\max}$ is the maximum delay (e.g., 60 seconds), and jitter is a random value in $[0, b)$ to desynchronize retries from multiple agents.

\subsection{Circuit Breaker}

If an agent fails consistently (e.g., fails 5 consecutive API calls), the circuit breaker opens: further calls to that agent are rejected immediately without attempting the API call. After a cool-down period (e.g., 30 seconds), the circuit transitions to half-open state, and one request is allowed. If it succeeds, the circuit closes and normal operation resumes. If it fails, the circuit opens again \cite{Segal2026}.

This pattern prevents cascading failures: a broken downstream agent stops accepting work, protecting upstream agents from unnecessary wait time and cost.

\section{Observability and Logging}\label{sec:logging}

Production systems are opaque without comprehensive logging. Every significant event must be recorded: agent invocation, token consumption, tool calls, errors, retries, and decisions.

Log entries must be machine-readable (JSON preferred) to enable downstream analytics, alerting, and cost reconciliation. Minimum fields for each log entry:

\begin{table}[h]
\centering
\begin{tabular}{p{3cm} p{5cm}}
\toprule
\textbf{Field} & \textbf{Description} \\[5pt]
\midrule
\texttt{timestamp} & ISO 8601 datetime when event occurred \\[5pt]
\texttt{agent\_id} & Identifier of agent that executed \\[5pt]
\texttt{task\_id} & Identifier of task being executed \\[5pt]
\texttt{event\_type} & \texttt{invoke}, \texttt{complete}, \texttt{error}, \texttt{retry} \\[5pt]
\texttt{tokens\_used} & Input and output token counts \\[5pt]
\texttt{status} & \texttt{success}, \texttt{failure}, \texttt{timeout} \\[5pt]
\texttt{error\_message} & Detailed error if status is failure \\[5pt]
\texttt{retry\_count} & Number of retry attempts before success \\[5pt]
\texttt{latency\_ms} & Execution time in milliseconds \\[5pt]
\bottomrule
\end{tabular}
\end{table}

With this instrumentation, operators can reconstruct the system's behavior, diagnose failures, and audit cost. Historical logs enable forecasting: given past token consumption per agent, estimate future costs and adjust budgets preemptively \cite{Segal2026}.

\section{Cost Optimization}\label{sec:costopt}

Token costs dominate multi-agent system expenses. A 15-page article might require 50k input tokens and 10k output tokens across all agents. At Claude 3.5 Sonnet pricing (\$0.003 per 1k input, \$0.015 per 1k output), one article costs approximately:

\begin{equation}
C = (50000 \times 0.003) + (10000 \times 0.015) = 150 + 150 = \$0.30
\label{eq:cost}
\end{equation}

At scale (100 articles per month), this grows to \$30 monthly. Several strategies reduce costs without sacrificing quality:

\subsection{Model Selection Strategy}

Reserve expensive, high-capability models (Claude 3.5 Opus) for complex reasoning tasks. Delegate routine tasks (formatting, citation assembly, simple editing) to cheaper models (Claude 3.5 Haiku). This tiered approach reduces average cost per task by 40--60\% while maintaining quality on critical steps \cite{LangChain2024}.

\subsection{Context Compression}

Instead of passing full article drafts between agents, pass summarized context. For example, the Writer receives not the entire research notes but a 200-word summary highlighting key claims. This reduces input tokens by 70--80\% with minimal information loss.

\subsection{Caching and Deduplication}

If the same research query is invoked multiple times, cache the result. Similarly, if two agents request similar summaries, serve from cache. Intelligent caching can reduce token costs by 30\% on typical workloads.

\subsection{Batch Processing}

Instead of generating one article at a time, queue 10 articles and have the Researcher process all of them in a batch request. Batch APIs from Claude offer 50\% token discounts, amortizing overhead across many requests \cite{Anthropic2024b}.

These strategies, applied together, reduce the per-article cost from \$0.30 to approximately \$0.12---a 60\% reduction---while maintaining output quality and throughput.
