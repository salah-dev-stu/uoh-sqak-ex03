\chapter{Orchestration Frameworks}\label{ch:frameworks}

Three dominant frameworks have emerged for multi-agent orchestration in production systems: CrewAI, LangChain, and LangGraph. Each makes distinct architectural choices optimized for different use cases and developer preferences. Additionally, Microsoft AutoGen provides an alternative approach emphasizing conversation-based orchestration. Understanding the strengths, limitations, and target scenarios of each framework is essential for architects designing multi-agent systems.

\section{LangChain and the Language Chain Expressiveness Language (LCEL)}\label{sec:langchain}

LangChain~\cite{LangChain2024a} provides a modular toolkit for building applications with language models. At its core, LangChain is not a prescriptive orchestration framework; rather, it is a composable library where developers assemble ``chains'' of components---prompt templates, LLM calls, parsers, and tools---into arbitrary data-flow graphs.

LangChain's philosophy centers on the \texttt{Runnable} interface, an abstraction that allows any component to be treated uniformly. A Runnable accepts input, transforms it, and produces output. Runnables compose via the pipe operator (\texttt{|}) or direct method calls. This composability enables fine-grained control: developers specify exactly how data flows between LLM invocations, tool calls, and custom logic.

The Language Chain Expressiveness Language (LCEL)~\cite{LangChain2024a} extends this paradigm, providing a syntax for declaring chains as composable, serializable data structures. LCEL chains can be debugged step-by-step, parallelized where independent branches exist, and logged exhaustively. This transparency is valuable in production debugging but comes at the cost of boilerplate: every orchestration step must be explicitly wired.

LangChain shines for applications requiring maximum flexibility and integration with diverse external systems---knowledge bases, APIs, databases, custom computation. The cost is a steeper learning curve and manual orchestration of control flow that more opinionated frameworks handle automatically.

\section{CrewAI Role-Based Agents}\label{sec:crewai}

CrewAI~\cite{CrewAI2024} takes a semantic, agent-centric approach. Rather than thinking about chains and data flow, developers define agents with explicit roles, goals, and backstories. Each agent encapsulates expertise, possesses access to a tool set, and receives tasks to execute.

The CrewAI \texttt{Agent} primitive is defined by:
\begin{itemize}
  \item \texttt{role}: a human-readable position (e.g., ``Senior Researcher'', ``Technical Writer'')
  \item \texttt{goal}: the agent's high-level objective
  \item \texttt{backstory}: narrative context that shapes the agent's behavior and reasoning style
  \item \texttt{tools}: a list of callable functions (web search, file I/O, domain-specific queries)
  \item \texttt{llm}: the language model provider (Anthropic, OpenAI, Groq) and parameters
\end{itemize}

CrewAI then organizes agents into a \texttt{Crew}, which manages task sequencing, dependency resolution, and output passing. The Crew primitive handles much orchestration boilerplate---retries, logging, state management---that developers must implement manually in LangChain.

A \texttt{Task} in CrewAI specifies:
\begin{itemize}
  \item \texttt{description}: the work to be done
  \item \texttt{expected\_output}: the format and type of the result
  \item \texttt{agent}: which agent executes the task
\end{itemize}

The semantic role system is CrewAI's greatest strength. By anchoring an agent to a narrative role and expertise context, the LLM adopts that persona, resulting in more consistent and task-appropriate behavior than generic prompt engineering. This makes CrewAI ideal for workflows with natural agent boundaries---research, writing, editing, production---where roles map directly to stages in a pipeline~\cite{CrewAI2024}.

\section{LangGraph Stateful Graphs}\label{sec:langgraph}

LangGraph~\cite{LangGraph2024} addresses a limitation of both LangChain and CrewAI: explicit state management. LangGraph treats orchestration as a state machine. Agents (or arbitrary nodes) process input, update shared state, and transition to the next node based on conditional logic.

A LangGraph state is a dictionary or structured object containing all mutable data---intermediate results, agent outputs, counters. Nodes are Python functions that read state, perform work, and return updated state. Edges are conditional branches that route execution based on the state. This explicit state representation enables:

\begin{itemize}
  \item \textbf{Serialization and Resumption}: state snapshots can be saved to disk or database, enabling workflow pause-and-resume across machine boundaries
  \item \textbf{Multi-Turn Loops}: iterative agent loops (agent deliberates, updates state, loops) are natural and explicit
  \item \textbf{Debugging}: developers inspect state at each node invocation, comparing inputs and outputs
  \item \textbf{Parallelization}: independent state updates can run concurrently, with synchronization at fan-in points
\end{itemize}

LangGraph is built on top of LangChain's primitives, leveraging the Runnable interface for compatibility. However, its state-machine model requires developers to think in graphs and explicit transitions, which has a steeper learning curve than CrewAI's role-based simplicity. LangGraph excels in scenarios demanding complex, iterative workflows---multi-turn dialogue, recursive problem decomposition, or agentic loops---where explicit state control provides safety and clarity~\cite{LangGraph2024}.

\section{Microsoft AutoGen}\label{sec:autogen}

Microsoft AutoGen~\cite{AutoGen2023} provides conversation-based orchestration, where agents interact via a message-passing abstraction. Rather than a central coordinator routing tasks, AutoGen agents exchange messages in a conversation, each agent deciding when to respond and what actions to take.

AutoGen defines:
\begin{itemize}
  \item \texttt{AssistantAgent}: an agent backed by an LLM, capable of code execution and tool use
  \item \texttt{UserProxyAgent}: a lightweight agent that simulates human interaction, approves code execution, or provides manual input
  \item \texttt{GroupChat}: a message pool shared by multiple agents; agents read the conversation history and decide when to contribute
\end{itemize}

The message-passing model is intuitive for collaborative problem-solving. Multiple agents brainstorm, debate, and refine solutions through conversation, without explicit orchestration rules. This mirrors human team dynamics and allows emergent behaviors that rigid task-sequencing frameworks constrain.

However, conversation-based orchestration sacrifices predictability. Without a central coordinator enforcing task order and deadlines, a GroupChat can loop indefinitely, diverge, or exhaust token budgets. AutoGen is well-suited to exploratory tasks, code generation with human review, and scenarios where emergent agent collaboration is the goal. It is less suitable for production pipelines with strict latency and cost constraints~\cite{AutoGen2023}.

\section{Framework Selection Criteria}\label{sec:selection}

Choosing among these frameworks requires matching the problem structure to the framework's strengths:

\textbf{Use LangChain} when building custom integrations with many external systems, the workflow does not fit standard patterns, or maximum transparency and debugging control are paramount. Accept increased development time and architectural responsibility.

\textbf{Use CrewAI} when the problem decomposes into natural agent roles, tasks are sequential or hierarchical with clear dependencies, and rapid development with semantic clarity is valued. Ideal for content production, research synthesis, and multi-stage workflows where roles map to pipeline stages.

\textbf{Use LangGraph} when the workflow is inherently iterative---agents loop, refine state, make conditional decisions---or when explicit state management and resumability are operational requirements. Preferred for complex agentic loops and stateful systems.

\textbf{Use AutoGen} when agent collaboration and emergent group intelligence are the primary goals, or when interactive human-in-the-loop approval is essential. Suits exploratory and creative tasks more than production pipelines.

For production deployments, all three frameworks require an external Gatekeeper~\cite{CrewAI2024} to enforce rate limits, token budgets, and safety constraints. The framework choice is architectural; the Gatekeeper is operational necessity.

The case study in Chapter~\ref{ch:casestudy} demonstrates CrewAI implementation for a structured content-production pipeline, where the role-based model aligns naturally with research, writing, editing, and LaTeX production stages. This architectural choice enables rapid development while maintaining semantic clarity, production-readiness, and ease of monitoring and debugging.
