\chapter{Case Study: This Article's Production Pipeline}\label{ch:casestudy}

\section{Overview}

This chapter documents the design, implementation, and operational characteristics of the multi-agent orchestration system that produced this article. The pipeline demonstrates practical application of CrewAI frameworks, LangChain tool integration, and LaTeX automation within a structured software engineering lifecycle. Rather than treating article generation as a monolithic task, the pipeline decomposes the work into specialized agent roles, each responsible for distinct phases of content creation, refinement, and publication formatting \cite{CrewAI2024}.

The architecture reflects lessons from production AI workflows: explicit task separation, measurable progress metrics, configurable resource constraints, and reproducible build artifacts. This case study serves both as a self-referential validation of the system design and as a template for similar multi-agent document generation projects \cite{LangChain2024a}.

\section{The Crew}

The production pipeline coordinates four primary agents, each with distinct role, goal, and backstory:

\textbf{Researcher Agent.} Responsible for gathering factual information, synthesizing source material, and producing detailed outlines. Uses web search tools to locate references and validates claims against authoritative sources. Generates raw research notes in Markdown format that feed downstream agents.

\textbf{Writer Agent.} Transforms research outlines into coherent prose chapters. Operates from the Researcher's output, maintaining stylistic consistency and ensuring logical flow between sections. Produces Markdown-formatted chapter drafts with placeholder sections for figures, tables, and formulas.

\textbf{Editor Agent.} Reviews drafts for clarity, grammar, consistency, and adherence to project constraints (e.g., chapter length targets, citation formatting). Provides structured feedback loops back to the Writer when revisions exceed a single pass. Outputs final Markdown ready for LaTeX conversion.

\textbf{LaTeX Agent.} Converts finalized Markdown chapters into valid LaTeX code, embeds TikZ diagrams for architecture sections, integrates figures and tables, and validates cross-reference syntax. Produces a complete \texttt{chapters/} directory tree ready for \texttt{lualatex} compilation with \texttt{biber} citation resolution \cite{LangChain2023}.

Each agent is instantiated with configurable model selection (defaulting to Claude via the Anthropic provider), rate-limited API calls through a Gatekeeper singleton, and dedicated logging channels for observability.

\section{Workflow Architecture}

The workflow follows a sequential-with-feedback pattern: Researcher produces outlines, Writer generates initial drafts, Editor validates quality, and LaTeX Agent produces compilation-ready LaTeX. If Editor feedback indicates major content gaps, the loop routes back to Writer; minor formatting issues proceed directly to LaTeX production.

Task definitions are loaded from JSON configuration files (\texttt{config/tasks.json}), allowing external tuning of agent prompts, tool selections, and output format specifications without code modification. The Crew processes execute against instantiated LangChain Tool objects: a web search tool, a file I/O tool for reading chapter templates, a chart generation tool that outputs PNG figures via matplotlib, and a LaTeX compilation orchestrator that invokes the system \texttt{lualatex} binary.

Control flow is managed by CrewAI's sequential task processor, which ensures dependency ordering and enforces timeout thresholds per task (defaulting to 300 seconds per agent invocation). Parallel execution of independent agents remains a future optimization; the current design prioritizes determinism and auditability.

\section{Key Engineering Decisions}

\textbf{Sequential over parallel task execution.} Upstream agent outputs (research notes) provide context for downstream agents (writer, editor). Parallelizing would require speculative chapter assignments, introducing risk of inconsistent style or redundant research. Sequential execution simplifies debugging and aligns with the iterative feedback loops inherent in document quality assurance.

\textbf{Markdown as intermediate representation.} LaTeX is verbose and error-prone for content authoring. Markdown-to-LaTeX conversion (via a dedicated tool within the LaTeX Agent) decouples content from typographic concerns, allowing Researcher and Writer agents to focus on substance while the LaTeX Agent handles \texttt{\%} escaping, bibliography key validation, and TikZ diagram embedding. This mirrors professional workflows used in projects like O\textquoteright{}Reilly Books and academic journal submissions.

\textbf{JSON configuration over code hardcoding.} Agent role, goal, and backstory strings; task descriptions; rate limit values; and model selections are all externalized to \texttt{config/} files. This satisfies the portability requirement: a grader can fork the repository, adjust \texttt{config/agents.json} to use their LLM provider, and execute the full pipeline without code changes \cite{Segal2026}.

\textbf{Gatekeeper for rate limiting and cost control.} All external API calls (LLM invocations, web search, LaTeX compilation) route through a singleton Gatekeeper that enforces per-agent token budgets and implements exponential backoff on rate-limit errors. This prevents runaway spend and ensures reproducible behavior across runs.

The metrics table documents both hard constraints (agent count, task count, LaTeX compilation passes) and quality targets (PDF length, test coverage, file modularity). Each cell reflects design trade-offs: the 150-line-per-file limit prevents cognitive overload and forces decomposition; the 85\% test coverage threshold balances mutation testing rigor with practical development velocity; the 4-pass LaTeX compilation ensures that bibliography citations, cross-references (\texttt{\textbackslash ref}), and table of contents numbers all stabilize before PDF generation.

This case study validates that the orchestration framework scales to a realistic document production workflow, with reproducible artifact generation, measurable quality metrics, and auditability via continuous commit history. The pipeline, once configured, executes deterministically: researchers can replicate the exact PDF on different machines (given equivalent LLM provider access and identical LaTeX installations) by checking out a git commit and running \texttt{uv sync} followed by the Makefile targets in the latex directory \cite{Microsoft2025}.
