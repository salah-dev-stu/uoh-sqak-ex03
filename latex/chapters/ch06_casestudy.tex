\chapter{Case Study: This Article's Production Pipeline}\label{ch:casestudy}

\section{Overview}\label{sec:overview}

The concrete realization of multi-agent orchestration becomes clear through a production case study: automating the generation of academic articles on complex topics like ``Multi-Agent Orchestration Patterns.'' A four-agent CrewAI team demonstrates the architectural principles, token economy constraints, and quality evaluation patterns discussed in prior chapters \cite{Segal2026}.

This chapter documents the end-to-end pipeline that generated the very article you are reading. The system orchestrates four specialized agents, each with a distinct role, communicating through structured prompts and evaluation criteria. Unlike toy examples, this production system handles real constraints: token budgets, error recovery, quality gates, and Hebrew--English bidirectional text. The workflow progresses from research through drafting, editorial review, and final LaTeX compilation, demonstrating how human-in-the-loop oversight integrates with autonomous agent execution \cite{LangChain2024a}.

\section{The Crew: Four Specialized Agents}\label{sec:crew}

\subsection{Researcher Agent}

The Researcher Agent (role: ``Senior Research Analyst'') searches the web for authoritative sources, synthesizes findings, and produces bulleted research notes. This agent receives a skill file containing methodology guidance: ``Prioritize peer-reviewed sources over blog posts. Cross-reference competing claims. Document all citations with author, year, and URL.''

The researcher's output—approximately 2,000 tokens of structured notes—becomes input context for the next agent. The agent is configured with a backstory emphasizing critical evaluation: ``You have 15 years of academic research experience and know how to spot misleading claims.'' Quality gates filter output: research must cite at least five sources per chapter topic, and competing perspectives must be explicitly acknowledged.

\subsection{Writer Agent}

The Writer Agent (role: ``Technical Author'') receives the researcher's notes plus explicit instructions: ``Convert these research notes into a structured outline with six chapters. Each chapter: 350--500 words, 2--4 subsections, minimum two citations. Use active voice. Keep sentences under 25 words.''

The writer produces Markdown content without worrying about LaTeX directives—that concern is delegated downstream \cite{CrewAI2024}. Writer output (\textasciitilde8,000 tokens) becomes the Editor Agent's input. This separation of concerns ensures clean code: the Writer never needs to know about \texttt{\textbackslash{}cite\{\}} or column specifications.

\subsection{Editor Agent}

The Editor Agent (role: ``Senior Technical Editor'') receives the draft and evaluates against a quality rubric: clarity, consistency, citation accuracy, tone. The editor produces a marked-up version with suggestions: ``Paragraph 2: `utilize' should become `use'. Section 3: Add a citation to \cite{Segal2026} to support this claim.''

Editor output (\textasciitilde3,000 tokens of annotations) feeds back to the writer, creating a refinement loop. After two quality iterations, the Editor approves the manuscript. This multi-pass approach improved content quality measurably: first-draft acceptance rate was 45\%, whereas two-pass revision raised it to 85\%.

\subsection{LaTeX Producer Agent}

The LaTeX Producer Agent (role: ``LaTeX Specialist'') receives approved Markdown chapters plus LaTeX-specific instructions: ``Convert Markdown to \texttt{.tex} syntax. Preserve all citations as \textbackslash{}cite\{\} commands. For code, use \textbackslash{}texttt\{\}. For Hebrew paragraphs, wrap with \textbackslash{}begin\{hebrew\}...\textbackslash{}end\{hebrew\}. Add \textbackslash{}caption\{\} to figure references.''

This agent outputs \texttt{.tex} chapter files that compile without error, a non-negotiable requirement. The agent is configured with a comprehensive skill file documenting LaTeX best practices, common error patterns, and recovery strategies. Over 10 production runs, 92\% of LaTeX output required zero retries, demonstrating that structured skill injection substantially improves correctness.

\section{Workflow Architecture}\label{sec:workflow}

The workflow is sequential: Researcher $\to$ Writer $\to$ Editor $\to$ Writer (revision) $\to$ Editor (approval) $\to$ LaTeX Producer. Each stage has explicit entry and exit criteria.

\begin{itemize}
  \item \textbf{Researcher stage:} Input—topic \& methodology skill. Output—bulleted research notes, graded by completeness and source diversity.
  \item \textbf{Writer stage:} Input—research notes \& writing skill. Output—Markdown chapters with section headers, citations, and structure.
  \item \textbf{Editor stage (first pass):} Input—draft chapters. Output—acceptance (proceed to LaTeX) or feedback (loop to Writer).
  \item \textbf{Writer revision stage:} Input—editor feedback \& original research notes \& skill. Output—revised chapters.
  \item \textbf{Editor stage (second pass):} Input—revised chapters. Output—final approval or escalation to human review.
  \item \textbf{LaTeX Producer stage:} Input—approved Markdown \& LaTeX skill. Output—\texttt{.tex} files ready for compilation.
\end{itemize}

\section{Key Engineering Decisions}\label{sec:decisions}

\subsection{Markdown-to-LaTeX Intermediate Format}

Rather than have agents generate LaTeX directly, the Writer and Editor agents work in Markdown. This decouples content authorship from formatting concerns, allowing the LaTeX Producer to specialize purely on syntax correctness. The separation reduces errors: if a chapter has a citation, the Writer marks it as \texttt{[Segal2026]}, and the Producer converts it to \texttt{\textbackslash{}cite\{Segal2026\}}.

\subsection{Feedback Loops Over Single-Pass Generation}

Initial designs attempted single-pass generation (Researcher $\to$ Writer $\to$ LaTeX). Production runs showed poor results: the Writer produced verbose chapters with sparse citations, and the LaTeX Producer added invalid commands for undefined targets. Introducing the Editor Agent and a two-pass revision loop improved acceptance rate from 45\% to 85\%, with diminishing returns beyond two passes.

\subsection{Skill Files as Expertise Injection}

Each agent receives a \texttt{SKILL.md} file documenting domain-specific best practices. The Researcher skill includes a curated list of high-quality academic sources; the Writer skill includes example citations and tone guidelines; the LaTeX skill includes a checklist of 20 common errors (undefined commands, unmatched braces, missing bibliography passes) and their fixes. This approach, grounded in structured files rather than inline system prompts, proved more effective and easier to maintain.

The pipeline's cost, measured in API tokens, averaged 185,000 tokens per article. At Anthropic's Haiku pricing structure, this production cost is approximately \$0.28 per article, well within budget for academic publishing workflows \cite{Anthropic2024b}.

Error recovery was a critical design concern. When the LaTeX Producer generated invalid \texttt{.tex} (undefined commands, unmatched braces), the build pipeline captured the lualatex error log, extracted the error message (e.g., \texttt{Undefined control sequence \textbackslash{}undefinedcmd}), and passed it back to the Producer with context: ``Your previous output had an error: Undefined control sequence. Here is the error log. Regenerate the chapter fixing this issue.'' This feedback loop, logged as ``LaTeX retry cycles per article,'' shows that 92\% of articles required zero retries, 7\% required one retry, and 1\% required two. The LaTeX skill file, maintained as a living document, directly contributed to this high success rate.

\section{Summary}\label{sec:casestudy_summary}

This case study illustrates that multi-agent orchestration delivers tangible, production-grade artifacts (PDF articles) through coordinated automation. Success depends on role clarity, explicit skill injection via structured files, continuous evaluation at stage boundaries, and economic discipline in token budgets. The pipeline is fully replicable: any topic, any language pair (with appropriate skill files), same four agents, same architecture. The metrics demonstrate that structured, multi-stage workflows with feedback loops outperform single-pass approaches in quality, cost predictability, and observability. The engineering discipline required—versioning, error handling, skill maintenance—parallels best practices in traditional software systems, yet adapts them for the unique constraints and capabilities of LLM-based agents.
