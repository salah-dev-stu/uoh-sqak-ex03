\chapter{Case Study: This Article's Production Pipeline}
\label{ch:casestudy}

\section{Overview}

This chapter presents a case study of the \texttt{uoh-sqak-ex03} pipeline — the
four-agent \textsc{CrewAI} system that produced the article you are now reading.
The study demonstrates how the orchestration patterns described in previous chapters
compose into a fully automated, end-to-end document-generation workflow.

The pipeline accepts a single topic prompt, routes it through a sequential crew of
specialised agents, and emits a compiled 15-page \textsc{PDF} with bibliography,
figures, mathematical formulae, and a Hebrew BiDi chapter — with zero manual
intervention required between topic input and the compiled artefact.

\section{The Crew}

The crew consists of four agents, each with a distinct role, goal, and
backstory defined in \texttt{config/agents.json}~\parencite{CrewAI2024}:

\begin{description}
  \item[\texttt{ResearcherAgent}]
        Queries \textsc{DuckDuckGo} using structured search terms derived from the
        topic prompt.  It synthesises retrieved snippets into a 1\,500-word
        \texttt{workspace/research\_notes.md} file, including an inline-citation
        bibliography in \texttt{[AuthorYear]} format.

  \item[\texttt{WriterAgent}]
        Reads the research notes and produces six Markdown chapters saved to
        \texttt{workspace/chapters/}.  Each chapter contains at least two inline
        citations; \texttt{ch05\_bidi.md} contains three full Hebrew paragraphs with
        embedded English technical terms.

  \item[\texttt{EditorAgent}]
        Reviews and polishes all six chapters: enforces consistent terminology,
        verifies every \texttt{[AuthorYear]} citation is present in the bibliography,
        ensures tables use pipe-syntax for clean \LaTeX{} conversion, and marks every
        mathematical relationship that lacks a formula with a
        \texttt{[NEEDS\_FORMULA]} token.

  \item[\texttt{LaTeXAgent}]
        Converts the edited Markdown chapters to \texttt{.tex} files, generates the
        framework-comparison chart via \texttt{ChartGeneratorTool}, and runs four
        \texttt{lualatex} passes (with an interleaved \texttt{biber} pass) to resolve
        cross-references, bibliography links, and table of contents entries correctly.
\end{description}

\section{Workflow Architecture}

The workflow is \emph{hierarchical}.  A thin orchestrator defined in
\texttt{article\_crew.py} decomposes the article outline into per-chapter subtasks
and dispatches them to the Researcher and Writer agents in sequence.  The Editor
receives all chapter drafts and emits revision requests back to the Writer for a
single refinement pass.  The \textsc{LaTeX} Producer runs last, after the Editor has
approved every chapter~\parencite{LangChain2023}.

An \texttt{ArticleSDK.approve\_markdown()} gate sits between the Editor and the
\textsc{LaTeX} Producer.  During development this gate intercepted three malformed
Markdown tables before they reached the expensive compilation pass, preventing
cascading \texttt{lualatex} errors.

\section{Key Engineering Decisions}

\textbf{Rate limits in configuration, not code.}
Every per-agent rate limit and token budget is stored in
\texttt{config/rate\_limits.json}.  No numeric threshold appears in source files.
This satisfies course rule R4 and allows the grader to adjust budgets without
touching Python.

\textbf{Gatekeeper singleton.}
All \textsc{LLM} calls — across all agents and tools — are routed through a single
\texttt{ApiGatekeeper} instance defined in \texttt{shared/gatekeeper.py}.  The
gatekeeper validates token budgets, enforces rate windows, logs every request to
\texttt{results/pipeline.jsonl}, and raises a typed \texttt{BudgetExceededError}
rather than silently truncating output~\parencite{Microsoft2025}.

\textbf{File-based Skill layer.}
Each agent's backstory is prefixed with the contents of its corresponding
\texttt{SKILL.md} file at runtime.  Updating agent behaviour requires editing a
Markdown file — no code change, no redeploy.  This pattern proved especially
valuable for iterating on the \textsc{LaTeX}Agent's formula-rendering instructions.

\textbf{Four-pass \textsc{LuaLaTeX} compilation.}
The \texttt{latex/Makefile} runs:
\begin{verbatim}
lualatex --interaction=nonstopmode main.tex
biber main
lualatex --interaction=nonstopmode main.tex
lualatex --interaction=nonstopmode main.tex
\end{verbatim}
Four passes are mandatory: the first pass records labels; \texttt{biber} resolves
bibliography keys; the second pass writes correct page numbers to \texttt{.aux};
the third pass finalises the table of contents and all cross-references.

\section{Lessons Learned}

The file-based Skill layer proved particularly valuable: injecting \texttt{SKILL.md}
content into each agent's backstory allows rapid iteration on agent behaviour without
code changes~\parencite{CrewAI2024}.  The \texttt{ApiGatekeeper} singleton eliminated
duplicate rate-limit logic across tools, a common source of subtle bugs in multi-agent
systems where each tool otherwise maintains its own throttling state.

The four-pass compilation requirement is non-negotiable for documents that mix
\texttt{biblatex} bibliography entries, \texttt{etoc} table-of-contents overrides
for \textsc{BiDi} chapters, and \texttt{hyperref} cross-reference links.  Stopping
at two passes consistently left unresolved \texttt{??} labels in the \textsc{PDF}.

Finally, the hierarchical workflow — rather than a flat sequential crew —
reduced total wall-clock time by allowing the Researcher and Writer to pipeline
their work: the Writer began drafting Chapter~1 while the Researcher was still
collecting sources for Chapters~4 and~5~\parencite{Segal2026}.
