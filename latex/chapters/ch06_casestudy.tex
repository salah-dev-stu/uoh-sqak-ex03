\chapter{Case Study: This Article's Production Pipeline}\label{ch:casestudy}

\section{Overview}

This chapter presents a real-world case study of multi-agent orchestration applied to technical article generation. The system demonstrates how CrewAI coordinates four specialized agents to produce a professionally formatted 15-page article on multi-agent orchestration, complete with cover sheet, table of contents, embedded images, Python-generated visualizations, mathematics, and bilingual (Hebrew-English) content. This case study validates the architectural principles discussed in previous chapters and illustrates practical engineering decisions required for production-quality output.

\section{The Crew: Four Specialized Agents}

The article generation pipeline consists of four core agents orchestrated through CrewAI \cite{CrewAI2024}:

\textbf{Researcher Agent}: Gathers authoritative information on multi-agent systems, CrewAI, LangGraph, and industry patterns. The Researcher accesses web search and document retrieval tools, compiling structured research notes with inline citations. This agent is configured with the role \texttt{Senior Research Analyst}, goal \texttt{gather comprehensive, cited information on multi-agent orchestration}, and backstory emphasizing 15+ years of systems research experience.

\textbf{Writer Agent}: Converts research notes into narrative Markdown chapters across six major topics: introduction, architectural patterns, framework comparison, production considerations, bilingual technical writing, and this production case study itself. The Writer produces approximately 2,200 words structured to balance depth with accessibility, ensuring each section includes at least two citations and clear transitions between topics.

\textbf{Editor Agent}: Reviews Writer output for clarity, consistency, terminology correctness, and technical accuracy. The Editor verifies that each citation reference exists in the research notes, checks chapter alignment with a unified voice, and flags sections requiring revision. This agent prevents hallucinated citations and ensures factual correctness before LaTeX conversion.

\textbf{LaTeX-Producer Agent}: Transforms validated Markdown into production-quality LaTeX source code compatible with LuaLaTeX and biber. This agent handles the most complex concerns: converting Markdown code blocks to \texttt{minted} environments, embedding Python-generated figures with proper scaling, rendering mathematics with \texttt{{\textbackslash}[...\textbackslash]} display mode, and implementing sophisticated BiDi (bidirectional) text handling for Hebrew paragraphs using \texttt{{\textbackslash}begin\{hebrew\}} directives and proper RTL markers. The LaTeX-Producer generates a Makefile that orchestrates four LuaLaTeX compilation passes with biber bibliography processing, ensuring all cross-references settle and citation links resolve correctly.

\section{Workflow Architecture}

The pipeline executes in a strictly sequential orchestration pattern:

\begin{enumerate}
\item \textbf{Research Phase}: The Researcher Agent gathers data, producing structured research notes indexed by topic and citation key.
\item \textbf{Writing Phase}: The Writer Agent receives research notes and generates six Markdown chapters, each 300--400 words, with embedded citation placeholders of the form \texttt{[AuthorYear]}.
\item \textbf{Editing Phase}: The Editor Agent validates each chapter, cross-references citations, and approves content for conversion.
\item \textbf{LaTeX Conversion Phase}: The LaTeX-Producer Agent converts approved Markdown to LaTeX, embeds figures, and generates compilation scripts.
\item \textbf{Compilation Phase}: External \texttt{make pdf} command invokes the Makefile, executing four LuaLaTeX + biber passes to produce the final PDF artifact.
\end{enumerate}

This sequential design reflects the document generation domain: later agents depend on outputs from earlier stages, and validation at each boundary prevents cascading errors.

\section{Key Engineering Decisions}

Three critical design decisions shaped this pipeline:

\textbf{Sequential Orchestration over Parallel}: Although CrewAI and LangGraph \cite{LangGraph2024} support task parallelization, document generation requires strict phase dependencies. A parallel Editor Agent cannot validate chapters that have not yet been written. Sequential execution trades latency for correctness.

\textbf{Markdown as Intermediate Representation}: Rather than asking the Writer Agent to produce LaTeX directly, the pipeline uses Markdown as an intermediate format. This choice decouples content generation (Writer's concern) from formatting (LaTeX-Producer's concern), allowing independent iteration and testing. Markdown is human-readable, version-control-friendly, and amenable to automated syntax checking.

\textbf{Configuration-Driven Prompts}: Each agent's role, goal, and backstory are stored in JSON configuration files, not hardcoded in Python. This enables rapid iteration on agent behavior without code changes. For example, modifying a chapter structure or citation style requires updating \texttt{agents.json}, not refactoring agent classes.
