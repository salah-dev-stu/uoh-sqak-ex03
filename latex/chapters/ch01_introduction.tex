\chapter{Introduction to Multi-Agent Orchestration}
\label{ch:introduction}

\section{Motivation}

Modern AI systems rarely succeed at complex, long-horizon tasks when designed as a single monolithic prompt.
A research article demands literature retrieval, synthesis, structured writing, editorial review, and precise
formatting — competencies that exceed what any single large language model (LLM) call can reliably deliver.
Multi-agent orchestration addresses this gap by distributing work across specialized autonomous agents, each
focused on a bounded subtask and equipped with domain-specific tools.

The discipline has deep roots in classical multi-agent systems (MAS) research \cite{WooldridgeJennings1995},
where theoretical foundations for agent communication, coordination, and emergent collective behavior were
established. What changed after 2022 is that LLMs proved they could serve as flexible, general-purpose
reasoning engines capable of planning, tool invocation, and self-correction \cite{WeiEtAl2022}. This
convergence — classical MAS theory meeting modern foundation models — produced today's practical
orchestration frameworks.

\section{What Multi-Agent Orchestration Is}

Multi-agent orchestration is the systematic coordination of multiple autonomous AI agents — each with
distinct roles, tools, and bounded context — toward a shared objective. Three architectural roles appear
in every system:

\begin{itemize}
  \item \textbf{Orchestrator}: decomposes the top-level goal, assigns subtasks, aggregates results, and
        decides when the pipeline terminates \cite{Anthropic2024b}.
  \item \textbf{Worker agent (subagent)}: receives a focused subtask, executes it using domain-specific
        tools, and returns a structured result.
  \item \textbf{Tool}: a callable function that extends an agent beyond its parametric knowledge — web
        search, code execution, file I/O, database access, or external API calls \cite{LangChain2024a}.
\end{itemize}

Agents also maintain \textbf{memory} across steps: in-context window memory for short-horizon tasks, and
external vector-database retrieval for long-horizon tasks where accumulated history would otherwise overflow
the context limit \cite{Park2023}.

\section{Why Decomposition Wins}

Empirical evidence consistently supports decomposition over monolithic prompting. On multi-step benchmarks,
specialist subagent pipelines outperform single-agent solutions \cite{Shinn2023}. The key insight is that
each agent operates in a focused context — it sees only the information relevant to its subtask, which
reduces hallucination and improves output quality. Role specialization also enables targeted tool access:
a researcher agent needs web search; a \LaTeX-producing agent needs a compile tool; neither benefits from
access to the other's tools.

\section{Scope of This Article}

This article surveys multi-agent orchestration patterns for technical practitioners building production AI
systems. Chapter~\ref{ch:architectures} covers the major architectural topologies. Chapter~\ref{ch:frameworks}
compares the leading frameworks. Chapter~\ref{ch:production} addresses production engineering concerns.
Chapter~\ref{ch:bidi} examines multilingual (BiDi) considerations. Chapter~\ref{ch:casestudy} presents a
concrete case study: an orchestrated pipeline that produces a fully formatted, typeset research article.

The goal is not to survey every framework but to identify the architectural primitives that recur across all
of them — the patterns a practitioner must understand to design, debug, and scale multi-agent systems in the
real world.
