\chapter{Agent Architectures and Topologies}\label{ch:architectures}

\section{Linear Sequential Architecture}\label{sec:linear-sequential}

The linear sequential topology is the simplest orchestration pattern: tasks execute in a fixed, predetermined order, and each task's output becomes the next task's input. This pattern is deterministic, easy to debug, and suitable for workflows where dependencies form a simple chain. In a linear pipeline, latency accumulates across stages; parallelism is zero because no two agents run simultaneously \cite{CrewAI2024}.

The linear model works well for content generation pipelines. A researcher agent completes its task, producing a research document. The writer agent consumes that document and produces a draft. The editor agent refines the draft. Finally, a LaTeX producer agent converts the polished markdown to PDF. Each stage has explicit input and output contracts, reducing ambiguity. The downside is latency: if any single stage takes long, the entire pipeline stalls. There is no opportunity for parallelism to hide latency \cite{LangChain2024a}.

\section{Hierarchical Manager-Worker Topology}\label{sec:hierarchical-manager}

Hierarchical orchestration introduces a central coordinator (the Manager) that delegates work to multiple worker agents. The Manager breaks a large task into subtasks, assigns each subtask to a worker, waits for all workers to complete, aggregates results, and moves to the next level of decomposition. This pattern enables partial parallelism: all workers at the same level run concurrently, though levels are sequential \cite{CrewAI2024}.

For instance, if a Manager task is ``research five historical developments in AI'', the Manager spawns five worker Researcher agents in parallel, each investigating one development. Once all five complete, the Manager aggregates their findings into a consolidated research document. The next level—Writing—begins only when the aggregation finishes. This pattern balances simplicity with parallelism and is the de facto standard in CrewAI's process design \cite{LangChain2024a}.

Hierarchical topology requires careful error handling. If one worker fails, the Manager must decide: retry the worker, skip the failed subtask, or abort the entire pipeline. These decisions are typically encoded in the process's exception handling logic.

\section{Peer-to-Peer Consensus Topology}\label{sec:peer-to-peer}

In a peer-to-peer topology, agents communicate directly without a central coordinator. Agents can route work to each other, negotiate on disagreements, and collectively converge on a solution. This topology is useful for debate and consensus scenarios: multiple specialist agents deliberate on a question, each offering a perspective, until agreement is reached or a voting threshold is met.

Peer-to-peer systems are highly resilient (no single point of failure) and naturally distributed. However, they risk deadlock (agents waiting for each other indefinitely) and require explicit termination conditions. In practice, peer-to-peer orchestration is implemented as a custom graph in LangGraph, where agents are nodes and message-passing edges define communication channels \cite{LangGraph2024}.

A concrete example: four specialist agents (Historian, Technologist, Ethicist, Economist) each evaluate a proposal from their perspective. They exchange critiques asynchronously until consensus is reached or a fixed number of rounds completes. The final agreed-upon judgment is the output.

\section{Tool-Augmented Specialists Topology}\label{sec:tool-augmented}

This topology equips each agent with a rich set of tools tailored to its specialty. Rather than agents calling other agents, they invoke domain-specific tools: a Researcher agent has web search, database query, and citation-parsing tools; a Writer agent has style-checking, paraphrasing, and grammar tools; an Editor agent has plagiarism detection and fact-checking tools. Tools are gated through a Gatekeeper to enforce rate limits and token budgets \cite{Anthropic2024b}.

Tool-augmented specialists allow each agent to solve its portion of the problem independently without directly depending on other agents' outputs (though inputs may be provided upfront). This topology scales well: new tools can be added without modifying agent logic, and tool invocations are fully observable in logs. The trade-off is that agents cannot leverage each other's reasoning in real time; tool results must be synthesized at a higher level.

\section{Topology Comparison}\label{sec:topology-comparison}

Each topology presents distinct trade-offs in latency, parallelism, complexity, and suitability. The latency of a linear sequential pipeline is the sum of all task latencies:

\begin{equation}
L_{\mathrm{seq}} = \sum_{i=1}^{N} \ell_i
\label{eq:pipeline_latency}
\end{equation}

where $\ell_i$ is the latency of stage $i$ and $N$ is the number of stages. In contrast, hierarchical topology latency is bounded by the critical path: the slowest worker at each level, plus coordination overhead.

Linear sequential pipelines are simplest but inflexible; they are appropriate when tasks have strict dependencies and latency is not critical \cite{CrewAI2024}. Hierarchical topologies sacrifice some of that simplicity to enable parallelism at each level; they are suitable for problems that decompose naturally into independent subtasks. Peer-to-peer consensus is costly in complexity but invaluable for scenarios where multiple perspectives must be reconciled. Tool-augmented specialists allow maximum independence but require careful tool design and gating to avoid inconsistent outputs \cite{LangChain2024a}.

For production multi-agent systems, CrewAI's default hierarchical orchestration strikes a pragmatic balance: enough parallelism to hide latency, enough centralization to ensure observability and error control. The article generation pipeline described in \autoref{ch:case-study} uses a linear sequential topology because the problem naturally decomposes into sequential stages: research informs writing, writing informs editing, editing informs publication. Non-linear topologies would add complexity without proportional benefit \cite{Segal2026}, \cite{Microsoft2025}.
