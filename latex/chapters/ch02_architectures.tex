\section{Sequential Pipelines}

\textbf{Sequential pipelines} pass a work artifact from one agent to the next in a fixed order.
Each agent transforms the artifact and hands it forward. The pattern is simple to reason about
and debug, but any single agent's failure blocks the entire chain \cite{LangGraph2024}.

For a pipeline of $N$ agents with individual latencies $\ell_1, \ell_2, \ldots, \ell_N$, the
total end-to-end latency satisfies

\begin{equation}
  L_{\mathrm{seq}} = \sum_{i=1}^{N} \ell_i + (N - 1)\,\delta
  \label{eq:seq_latency}
\end{equation}

where $\delta$ denotes the artifact-transfer overhead between consecutive agents. The additive
dependence on $N$ motivates shifting independent subtasks to parallel topologies whenever
the task graph permits.

\section{Hierarchical Manager--Worker Architectures}

\textbf{Hierarchical (manager--worker) architectures} introduce an orchestrator agent that
decomposes a goal into subtasks and dispatches them to worker agents. The orchestrator collects
results, evaluates quality, and either accepts or re-dispatches \cite{CrewAI2024}. This mirrors
how human organisations delegate work; both CrewAI and AutoGen implement this pattern as a
first-class construct \cite{AutoGen2023}.

\section{Peer-to-Peer Meshes}

\textbf{Peer-to-peer meshes} allow agents to call one another directly without a central
co-ordinator. Emergent behaviour is possible, but observability and deadlock prevention become
significant engineering challenges. The absence of a single control point makes formal
verification of liveness properties considerably harder.

\section{Hybrid Orchestrators}

A hybrid approach---a thin orchestrator managing loosely coupled specialist agents---balances
control with flexibility and is the dominant pattern in production systems today
\cite{LangGraph2024}. The orchestrator retains global state and task-routing logic while each
worker operates independently, enabling partial retries without replaying the full pipeline.

\section{Cost and Topology Selection}

Choosing a topology is an early architectural decision with long-term consequences for
testability, latency, and cost. The expected inference cost $C$ of a manager--worker pipeline
with $K$ parallel workers is

\begin{equation}
  C = \sum_{k=1}^{K}\!\bigl(c_{\mathrm{in}}\,t_{k}^{\mathrm{in}}
      + c_{\mathrm{out}}\,t_{k}^{\mathrm{out}}\bigr) + C_{\mathrm{orch}}
  \label{eq:cost_model}
\end{equation}

where $c_{\mathrm{in}}$ and $c_{\mathrm{out}}$ are the per-token input and output prices,
$t_{k}^{\mathrm{in}}$ and $t_{k}^{\mathrm{out}}$ are the token counts for worker~$k$, and
$C_{\mathrm{orch}}$ is the orchestrator's own inference cost \cite{Segal2026}. Minimising $C$
under a quality constraint is the central optimisation problem of production multi-agent
engineering.
