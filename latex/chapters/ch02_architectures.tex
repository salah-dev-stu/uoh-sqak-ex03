\chapter{Agent Architectures and Topologies}\label{ch:architectures}

\section{Linear Sequential Topology}

In a linear sequential topology, agents execute tasks in strict order where each agent completes its work before the next begins. This architecture proves ideal for content generation pipelines where outputs from earlier stages directly inform later stages. The researcher produces findings, which the writer transforms into prose, which the editor refines for clarity and correctness, which the LaTeX producer formats into publication-ready output \cite{CrewAI2024}.

\begin{figure}[H]\centering\resizebox{\textwidth}{!}{\input{figures/crew_architecture.tikz}}\caption{CrewAI multi-agent architecture.}\label{fig:crew_architecture}\end{figure}

The sequential model minimizes context switching. Each agent operates on a complete, coherent input generated by its predecessor. This reduces cognitive overhead compared to interleaved or branching execution where agents must reason about partial or competing states. Token efficiency improves because intermediate outputs serve as training signals for downstream agents—the researcher's findings become explicit context rather than being re-encoded through independent queries \cite{Segal2026}.

Sequential execution carries a latency penalty. If each of $N$ agents requires time $\ell_i$ to complete their task, total execution time is:

\begin{equation}
L_{\mathrm{seq}}=\sum_{i=1}^{N}\ell_i\label{eq:pipeline_latency}
\end{equation}

This contrasts sharply with parallel execution, where the system could theoretically invoke all agents simultaneously and wait only for the slowest—but only if their tasks are independent, which content generation tasks rarely are. The researcher must complete before the writer can write meaningfully; the editor needs the draft before editing \cite{CrewAI2024}.

\section{Hierarchical Manager-Worker Topology}

Hierarchical orchestration introduces a manager agent that sits above domain-specific workers. The manager receives a high-level goal, decomposes it into subtasks, assigns each to a specialist agent, monitors progress, and synthesizes results. This topology shines when task decomposition is non-obvious or when intermediate decisions influence subsequent steps \cite{CrewAI2024}.

Consider a manager agent tasked with producing a comprehensive article on machine learning interpretability. The manager might decompose this into: research task gathering papers on LIME and SHAP; writer task drafting sections on each technique with comparative analysis; editor task reviewing for internal consistency; evaluator task ensuring claims cite original papers. If the evaluator identifies gaps, the manager may loop back to the researcher with refined queries \cite{Segal2026}.

The manager-worker topology enables dynamic orchestration. Unlike sequential systems with fixed task ordering, hierarchical systems adapt. The manager observes intermediate outputs and decides next steps accordingly. This flexibility proves essential when task structure depends on content characteristics—a research article on a novel topic may require deeper investigation; one on an established topic may skip exploratory search \cite{CrewAI2024}.

\section{Peer-to-Peer Consensus Topology}

Peer-to-peer topologies emphasize consensus among independent agents rather than centralized control. Multiple specialist agents evaluate the same problem from different angles, and a consensus mechanism aggregates their perspectives. This architecture mitigates single-agent failure modes and leverages diverse reasoning patterns.

A three-agent peer consensus might include: Research Agent focusing on factual accuracy and source credibility; Clarity Agent optimizing for reader comprehension and pedagogical structure; Innovation Agent seeking novel connections and forward-looking analysis. All three evaluate the same draft independently, then their feedback is synthesized into revision guidance \cite{Segal2026}.

Peer consensus excels at validation. When agreement exists across diverse perspectives, confidence in correctness rises. Conversely, disagreement signals ambiguity that humans should resolve. This makes peer-to-peer suitable for high-stakes content like academic papers where conservative aggregation proves safer than any single agent's output \cite{Wooldridge2009}.

\section{Tool-Augmented Specialists Topology}

Tool-augmented specialists combine agent capabilities with external tools, enabling rich environmental interaction. Rather than agents reasoning purely from context windows, they invoke web search, databases, code execution, and file I/O. This topology proves essential for information-retrieval intensive tasks and for generating artifacts like charts and code.

In an article-writing system, the Researcher agent accesses web search tools to discover recent publications and bibliography tools to build citations. The Writer agent uses document templates and formatting tools. The Editor agent invokes spelling tools and readability metrics. The LaTeX Producer agent invokes compilation tools to validate structure and generate PDFs \cite{CrewAI2024}.

Tools introduce attack surfaces and latency. External tool invocations depend on network availability and API reliability. A Gatekeeper class must mediate all tool calls, enforcing rate limits, token budgets, and permission checks. Tool responses may be malformed or misleading, requiring output validation before downstream use \cite{Segal2026}.

The topology selection depends on problem structure, latency constraints, and risk profile. Sequential excels for deterministic pipelines with clear dependencies. Hierarchical adapts when task structure depends on intermediate results. Peer-to-peer validates through diverse perspectives. Tool-augmented enables rich environmental interaction. Production systems typically employ hybrid approaches, nesting peer consensus within hierarchical delegation or running parallel peer reviews on sequential pipeline outputs \cite{CrewAI2024}.

Context passing efficiency also differs across topologies. Sequential systems minimize context overhead by passing outputs directly as inputs—the researcher's findings become the writer's instructions without re-encoding. Hierarchical systems require the manager to maintain state across multiple parallel worker executions, increasing manager's memory burden. Peer systems must encode the target artifact multiple times, once per peer. Tool-augmented systems incur overhead from tool output parsing and reformatting \cite{Segal2026}.

Recovery and fault tolerance vary significantly. Sequential systems fail catastrophically—if the researcher produces hallucinations, all downstream work propagates errors. Hierarchical systems allow the manager to detect quality issues and loop back to fix earlier stages. Peer systems catch errors through disagreement, surfacing ambiguous decisions for human review. Tool-augmented systems depend on tool reliability; graceful degradation requires fallback paths when tools fail \cite{Wooldridge2009}.
