\section{Linear Sequential Architecture}

The simplest multi-agent topology chains agents in sequence, where each agent's output becomes the next agent's input. This research-write-edit pipeline is canonical in document generation: a researcher agent gathers information and sources, a writer agent synthesizes this research into prose, and an editor agent polishes the output for publication \cite{CrewAI2024}.

Linear architectures excel for problems with clear sequential dependencies and well-defined handoff points. Each agent can focus on its specialized task without coordinating with others. However, sequential execution inherently limits parallelism---wall-clock time grows linearly with the number of stages, and any agent failure blocks downstream work. Token consumption is also predictable but potentially high, since each subsequent agent must consume prior outputs to maintain context \cite{LangChain2024a}.

The latency of a sequential pipeline with $N$ agents is given by:
\begin{equation}
L_{\mathrm{seq}} = \sum_{i=1}^{N}\ell_i
\label{eq:pipeline_latency}
\end{equation}
where $\ell_i$ represents the execution latency of agent $i$. This linear growth in latency motivates the hierarchical and parallel topologies discussed below.

\section{Hierarchical Manager-Worker Architecture}

Larger systems often employ hierarchical structures where a supervisor agent coordinates multiple specialist teams. The supervisor decomposes high-level goals into team assignments, monitors progress, arbitrates conflicts, and escalates blockers. This pattern scales well to large agent networks---the supervisor maintains responsibility for big-picture coordination while specialist teams focus on domain-specific execution.

Hierarchical architectures introduce communication overhead. The supervisor must maintain context across multiple teams, delegate intelligently, and synthesize results. Failure at the supervisor level cascades across the entire system. However, this structure mirrors human organizations and provides clear authority and decision-making paths, making orchestration logic more transparent \cite{CrewAI2024}.

\section{Peer-to-Peer and Consensus Architectures}

For high-stakes decisions or complex reasoning, peer-to-peer architectures allow multiple agents to work independently on the same problem, then reach consensus through discussion or judgment. Each agent generates independent analysis, and a meta-agent evaluates proposals, identifies agreements and disagreements, and escalates controversial areas.

This approach leverages diverse reasoning styles to catch errors and oversights but multiplies token consumption---if five agents independently analyze a problem, the system pays five times the token cost. Consensus architectures are most valuable for critical decisions (technical architecture, risk assessment, content moderation) where the additional cost is justified by higher confidence in the outcome \cite{Anthropic2024b}.

\section{Tool-Augmented Specialist Topologies}

Agents with access to specialized tools (code execution, scientific APIs, database queries, web search) can solve problems beyond pure reasoning. A code-generation agent becomes more capable when it can execute and test code. A data-analysis agent with SQL and statistical packages can verify hypotheses against real data. A research agent with academic database access can cite authoritative sources \cite{LangChain2024a}.

Tool augmentation shifts the bottleneck from reasoning capability to tool availability and security. Specialized tools enable concrete outputs but require careful sandboxing to prevent misuse. Tool integration also introduces new failure modes---API rate limits, authentication timeouts, tool execution failures---that orchestration must handle gracefully.

\section{State Management Across Topologies}

Regardless of topology, state management is critical. In linear architectures, state flows forward---researcher findings become writer input. In hierarchical systems, state must be available at multiple levels---supervisors need summaries while specialist teams need detailed context. In peer-to-peer architectures, state consistency across agents is challenging; different agents may have contradictory understandings until consensus is reached.

Explicit state management (as in \textit{LangGraph}) makes these differences visible and testable. Implicit state management (as in simpler frameworks) requires careful architectural discipline to prevent information loss, state inconsistency, or excessive context accumulation. The choice of topology should be informed by state management complexity and the system's tolerance for eventual consistency versus strong agreement \cite{LangGraph2024}.

\section{Architectural Selection Guidelines}

Selecting the optimal topology requires balancing latency, parallelism, complexity, and problem characteristics. Sequential topologies minimize architectural complexity and work well for tightly coupled workflows with clear handoffs. Hierarchical topologies scale to larger agent teams but introduce supervisor overhead. Peer-to-peer topologies maximize confidence in critical decisions at the cost of token efficiency and coordination complexity. Tool-augmented topologies enable concrete problem-solving by grounding agents in external verifiable data sources.

The decision tree is straightforward: if tasks have clear sequential ordering and tight coupling, choose sequential; if the problem naturally decomposes into independent teams, choose hierarchical; if correctness matters more than cost, choose peer-to-peer with consensus; if grounding and verification are essential, augment selected agents with specialized tools \cite{CrewAI2024}.
